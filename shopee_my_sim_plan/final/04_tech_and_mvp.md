# 最终版 04：技术方案与MVP开发计划

## 1. 技术目标
先做“可运行经营系统”，UI从简，确保：
- 订单模拟可信
- 财务结算准确
- 100人联机稳定

## 2. 建议架构
- 前端：React + TypeScript
- 后端：FastAPI（Python）
- 数据：PostgreSQL + Redis
- 任务：异步队列处理订单流与周结
- 部署：Docker Compose

## 3. 服务边界
- `market-service`：市场指数、节庆、大盘流量
- `order-service`：订单生成、状态流转
- `inventory-service`：库存、补货、仓配约束
- `finance-service`：利润、现金流、汇率、报表
- `event-service`：事件触发与连锁
- `league-service`：联机赛季与排行榜

## 4. MVP最小可用范围
必须实现：
1. 多品类 + SKU库存
2. 自动下单引擎
3. 广告与价格策略
4. 利润与现金流结算
5. 单机时间系统（3x/5x/10x）
6. 联机100人同赛季基础版

## 4.1 前端页面范围（1.0）
- `/dashboard`：经营总览
- `/products`：商品与定价
- `/supply-inventory`：采购与库存
- `/logistics`：物流与仓配
- `/marketing`：广告与活动
- `/service`：客服与售后
- `/finance`：财务与回款
- `/market-view`：市场镜像（买家视角）

说明：
- 1.0采用“卖家端主流程 + 买家视角镜像页”架构，不实现完整买家端账号与支付流程。
- 市场镜像用于联机竞品观察与策略对标，是强对抗玩法的必要界面。

## 5. 12周计划（建议）
1. 周1-2：数据结构与基础服务骨架
2. 周3-4：订单与库存链路跑通
3. 周5-6：广告、价格、转化模型
4. 周7-8：财务、汇率、融资
5. 周9-10：联机赛季、排行榜
6. 周11-12：压测、平衡、修复

## 6. 压测指标
- 100人同服，订单模拟按分钟批处理
- 关键接口P95 < 300ms
- 周结任务成功率 > 99.5%

## 7. 开发优先级
- P0：经济闭环正确
- P1：联机对抗稳定
- P2：交互体验与视觉升级
