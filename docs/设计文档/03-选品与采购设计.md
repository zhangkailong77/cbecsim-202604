# CBEC SIM 阶段设计文档（第三步）

## 阶段进度
- 第一步（已完成）：注册/登录功能
- 第二步（已完成）：开局准备功能
- 第三步（当前）：选品与采购功能设计

## 第三步目标
- 玩家在“销量榜 / 新品榜 / 热推榜”中自由选品。
- 采购单位按件，资金单位统一为人民币（RMB）。
- 玩家下单后即时扣减可用资金，沉淀采购订单数据供后续环节使用。

## 本阶段定位
- 本阶段只做“选品与采购”本身，不做结算复盘。
- 玩家可在现金足够前提下无限次采购。
- 玩家依据三榜信息自主判断，不提供系统推荐答案。

## 页面交互设计（V1）
1. 页面布局
- 左侧：榜单切换（`销量榜`、`新品榜`、`热推榜`）
- 中间：商品列表（分页展示，每页 `20` 条）
- 右侧：采购面板（购物车）

2. 商品行信息
- 展示字段：商品名、类目、采购价（RMB/件）、建议售价、榜单指标（销量/增速/热度）
- 行操作：输入采购数量（默认 `1`）+ 点击“加入采购”
- 数量校验：必须为正整数

3. 采购面板
- 展示已选商品：单价、数量、小计
- 支持操作：修改数量、删除、清空
- 实时展示：
  - 采购总额
  - 下单后余额（`当前可用资金 - 采购总额`）
- 若总额超出资金：禁止提交并提示“超出可用资金”

4. 提交采购
- 点击“确认采购”后：
  - 扣减可用资金
  - 生成采购订单与明细
  - 刷新本局累计采购数据
- 失败处理（资金不足/并发冲突）：提示错误并刷新余额

## 后端数据结构（V1）
1. 商品池表（已有，扩展字段）
- `market_products`
- 关键字段：
  - `id`
  - `market`
  - `board_type`（`sales | new | hot`）
  - `category`
  - `product_name`
  - `supplier_price`
  - `suggested_price`
  - `monthly_sales`
  - `growth_rate`
  - `hot_score`

2. 游戏局表（已有）
- `game_runs`
- 关键字段：
  - `id`
  - `user_id`
  - `status`
  - `initial_fund`

3. 采购订单主表（新增）
- `procurement_orders`
- 字段：`id, run_id, user_id, total_amount, created_at`

4. 采购订单明细表（新增）
- `procurement_order_items`
- 字段：`id, order_id, product_id, product_name_snapshot, unit_price, quantity, line_total`

## 接口设计（V1）
1. `GET /market/leaderboard`
- 入参：`board_type, category, keyword, page, page_size=20`
- 出参：商品分页列表、总条数

2. `GET /game/runs/{run_id}/procurement/cart-summary`
- 返回：可用资金、累计采购额

3. `POST /game/runs/{run_id}/procurement/orders`
- 入参：`items: [{ product_id, quantity }]`
- 服务端处理：
  - 读取商品采购价
  - 计算订单总额
  - 校验资金是否足够
  - 扣减 `cash_balance`
  - 事务写入订单主表与明细
- 出参：`order_id, total_amount, cash_balance_after`

4. `GET /game/runs/{run_id}/procurement/orders`
- 返回当前局采购历史

## 规则细节（V1）
1. 三榜口径
- 销量榜（`sales`）：按 `monthly_sales` 降序
- 新品榜（`new`）：按 `new_score` 降序
- 热推榜（`hot`）：按 `hot_score` 降序
- 当前仅支持市场 `MY`

2. 类目范围
- 仅保留三个类目：`美妆个护`、`手机与数码`、`服饰配件`
- 不包含“店铺类型”筛选

3. 同商品跨榜处理
- 同一商品可出现在多个榜单
- 唯一标识以 `product_id` 为准
- 重复加入同商品时，购物车合并数量，不新增行

4. 金额与数量约束
- 金额单位：`RMB`（整数）
- 数量单位：件（正整数）
- 行小计：`unit_price * quantity`
- 订单总额：各行小计求和
- 资金约束：`cash_balance_after >= 0`，否则拒单

5. 一致性与并发
- 下单接口使用数据库事务
- 余额扣减使用原子条件或行锁，防止并发超扣
- 前端提交期间按钮进入 loading，防止重复点击

## 本阶段暂不包含
- 库存周转天数
- 在途库存流转（采购后转在途再到货）
- 结算复盘逻辑

## 验收标准
- 玩家可在三榜自由切换并选品。
- 每页固定展示 `20` 条并支持翻页/快速跳转。
- 玩家可无限次采购，但每次下单都必须通过资金校验。
- 提交成功后资金正确扣减，订单与明细持久化成功。
- 刷新页面后可恢复本局采购历史与最新余额。
